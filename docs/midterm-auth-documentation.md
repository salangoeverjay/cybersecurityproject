# Cybersecurity Midterm Project Documentation

## Project Title
Secure User Authentication System using Laravel + MySQL (Laragon)

## 1. Overview
This project implements a manual authentication flow in Laravel without relying on default Laravel auth scaffolding.  
It includes:
- User registration
- User login
- Session-based authenticated page (dashboard)
- Secure password handling using **salt + pepper + SHA-256**

The credentials table stores only:
- `username`
- `password_hash`
- `salt`

The system never stores plain text passwords.

## 2. Hashing Algorithm Used
The project uses the SHA-256 hashing algorithm through PHP's built-in `hash()` function:

```php
hash('sha256', $password . $salt . $pepper)
```

### Why SHA-256?
- It is a widely used cryptographic hash algorithm.
- It produces a fixed-length, one-way digest (64 hex characters).
- It is suitable for demonstrating secure hashing logic in coursework.

Note: In production systems, adaptive password hashing algorithms such as `password_hash()` with bcrypt/Argon2 are recommended. This midterm implementation intentionally shows explicit salt and pepper handling for learning objectives.

## 3. How Salt Is Generated
Each user receives a unique salt generated by:

```php
bin2hex(random_bytes(16))
```

### Security properties
- `random_bytes(16)` is cryptographically secure.
- Each registration generates a different salt, even for the same password.
- Salt is stored in the database because it is needed during login hash recreation.

## 4. Where Pepper Is Stored
Pepper is stored in environment configuration:
- `.env` key: `PASSWORD_PEPPER=...`
- Loaded in code via: `config('security.password_pepper')`
- Config location: `config/security.php`

Pepper is **not stored in the database**.

### Why this matters
If an attacker steals only the database, they still do not have the pepper secret.  
Without the pepper, precomputed cracking attempts become significantly harder.

## 5. Registration Security Flow
1. User submits `username` and `password`.
2. Input is validated (required, length checks, username format, unique username).
3. Server generates random salt.
4. Server reads pepper from environment.
5. Server computes hash: `SHA-256(password + salt + pepper)`.
6. Database stores only `username`, `password_hash`, `salt`.
7. User sees registration success message.

## 6. Login Security Flow
1. User submits `username` and `password`.
2. Server validates input format.
3. Server fetches user record by username.
4. Server reads stored salt and environment pepper.
5. Server recomputes hash from input password.
6. Server compares stored hash and computed hash using:

```php
hash_equals($storedHash, $computedHash)
```

7. If matched: login success and session regeneration.
8. If not matched: `Invalid Username or Password`.

## 7. Why Salt + Pepper Improves Security
### Salt benefits
- Prevents identical passwords from having identical hashes.
- Defeats rainbow table reuse across users.

### Pepper benefits
- Adds a secret value outside the database.
- Protects against offline cracking when DB is leaked alone.
- Forces attacker to also compromise application configuration.

Using both gives stronger defense-in-depth than hashing with password alone.

## 8. Additional Security Practices in This Project
- No plain text password storage.
- Server-side input validation.
- Unique username constraint in validation and database schema.
- SQL injection protection via Eloquent/Query Builder and parameterized queries.
- Secure hash comparison with `hash_equals`.
- Session ID regeneration on successful login to reduce session fixation risk.

## 9. Database Structure Summary
Main credentials table:
- `id` (bigint, primary key)
- `username` (varchar, unique)
- `password_hash` (varchar(64))
- `salt` (varchar(64))
- `created_at` (timestamp)
- `updated_at` (timestamp)

This schema satisfies the requirement to store only username, hash, and salt for user credentials.
